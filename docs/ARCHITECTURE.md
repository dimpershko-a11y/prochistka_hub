# Архитектура PRO-CHISTKA Hub

## Цель

PRO-CHISTKA Hub должен стать единым рабочим приложением, но внутри оставаться набором независимых модулей.

Главная задача архитектуры — позволить менять один раздел без риска сломать остальные.

## Слои проекта

```txt
src/
  app/       # ядро приложения
  modules/   # бизнес-модули
  shared/    # общий код
  config/    # глобальная конфигурация
  assets/    # статические файлы
```

## app

Слой `app` отвечает только за запуск приложения:

- оболочка интерфейса;
- меню;
- роутинг;
- регистрация модулей;
- загрузка настроек;
- подключение общих стилей.

В `app` нельзя помещать бизнес-логику калькулятора, чек-листов, CRM, табеля и других разделов.

## modules

Каждый модуль отвечает за один раздел бизнеса.

Модуль может иметь:

```txt
src/modules/module-name/
  index.js        # публичная точка входа
  manifest.js     # описание модуля для Hub
  components/     # UI-компоненты модуля
  services/       # бизнес-логика модуля
  models/         # структуры данных модуля
  styles.css      # стили модуля
```

Модуль не должен напрямую импортировать внутренние файлы другого модуля.

Разрешено:

- импортировать функции из `shared`;
- отдавать наружу публичные методы через `index.js`;
- обмениваться событиями через общий слой событий;
- использовать общий API-клиент.

Запрещено:

- менять данные другого модуля напрямую;
- дублировать общие расчёты;
- хранить общие настройки внутри модуля;
- смешивать PDF-генерацию с бизнес-логикой модуля, если PDF нужен нескольким разделам.

## shared

В `shared` лежат общие сервисы:

```txt
src/shared/
  api/             # API-клиенты и запросы
  calculations/    # общие расчёты
  events/          # обмен событиями между модулями
  formatters/      # деньги, даты, телефоны, площади
  google-sheets/   # работа с Google Sheets
  pdf/             # генерация PDF
  settings/        # чтение/запись настроек
  storage/         # localStorage/sessionStorage/будущее хранилище
  ui/              # общие UI-хелперы
```

## Контракт модуля

Каждый модуль должен экспортировать объект `moduleManifest` и функции `mount(container, context)` / `unmount()`.

## Контекст модуля

Hub передаёт модулю `context`:

```js
{
  shared,
  settings,
  eventBus,
  navigate
}
```

Это позволяет модулю работать через общие слои, не привязываясь к внутренностям приложения.

## Правило изменения

Перед изменением любого раздела нужно определить:

1. Это логика конкретного модуля?
2. Это общая функция, которая понадобится нескольким модулям?
3. Это настройка всего Hub?

Если функция общая — она должна быть в `shared`.
Если функция относится к одному разделу — она остаётся внутри модуля.
